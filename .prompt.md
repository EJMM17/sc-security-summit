---
title: Actualiza archivos locales desde una repo de GitHub
summary: |
  Prompt reutilizable para actualizar/mergear contenido local desde un repositorio GitHub remoto.
  Diseñado para un agente que ejecuta operaciones Git asistidas por el usuario con opciones de seguridad y modo `dry-run`.
---

Propósito
- Ayudar a sincronizar archivos locales con una rama remota de GitHub, ofreciendo pasos claros, opciones seguras y salidas estructuradas.

Entradas (parámetros)
- `repo_url` (string, requerido): URL remota (https o ssh) del repo GitHub.
- `branch` (string, opcional, default: `main`): rama remota a traer.
- `path` (string, opcional): ruta o patrón (glob) para limitar los archivos a actualizar.
- `strategy` (string, opcional): `merge` | `rebase` | `reset-hard` | `fetch-only` (default: `merge`).
  - `preserve_local` (string, opcional): `stash` | `abort` | `overwrite` (default: `overwrite`).
- `create_backup` (boolean, opcional, default: true): si crear copia de seguridad antes de sobrescribir.
- `dry_run` (boolean, opcional, default: true): no aplicar cambios, solo mostrar acciones previstas.
- `commit_local_changes` (boolean, opcional): si el agente debe auto-commit de cambios locales antes de sincronizar.

Salida esperada
- `status`: `ok` | `conflicts` | `error`.
- `summary`: breve descripción de las acciones realizadas.
- `commands`: lista de comandos Git que el agente ejecutó o propondrá.
- `changed_files`: lista de rutas afectadas con tipo `updated`|`added`|`deleted`.
- `diffs`: links o fragmentos de diff por archivo (cuando sea posible).
- `conflicts`: si hay conflictos, lista de archivos y sugerencia de resolución.

Seguridad y buenas prácticas
- Nunca muestres credenciales ni claves en texto plano.
- Si `preserve_local=overwrite`, pedir confirmación explícita antes de proceder.
- Recomendar `dry_run=true` por defecto para repos grandes o cambios destructivos.

Comportamiento detallado (lógica que debe seguir el agente)
1. Validar `repo_url` y `branch`.
2. Hacer `git fetch origin <branch>`.
3. Evaluar diferencias entre `HEAD` y `origin/<branch>` (o entre `path` si se especificó).
4. Si `dry_run=true`, devolver `commands` y `summary` sin aplicar.
5. Si `dry_run=false` y `preserve_local=stash`, ejecutar `git stash push -m "auto-stash before sync"` antes de aplicar.
6. Aplicar la estrategia:
   - `merge`: `git merge --no-ff origin/<branch>`
   - `rebase`: `git rebase origin/<branch>`
   - `reset-hard`: `git reset --hard origin/<branch>` (DEStructivo: pedir confirmación)
   - `fetch-only`: solo mostrar diffs
7. Si hay conflictos, parar y devolver `status: conflicts` con instrucciones de resolución.
8. Si `create_backup=true`, crear un archivo zip con los archivos que se van a sobrescribir y devolver su ruta.

Ejemplos de invocación (texto que el usuario puede dar)
- "Actualiza los archivos locales con la repo `https://github.com/org/proyecto.git`, rama `develop`, haz un `dry_run` primero."
- "Sincroniza `components/` desde `git@github.com:org/proyecto.git` rama `main`, preserva cambios locales con `stash` y aplica `merge` (no dry-run)."

Plantilla de prompt para el agente
---
Tarea: "actualiza los archivos locales con la repo de GitHub"
Parámetros:
- repo_url: {{repo_url}}
- branch: {{branch|main}}
- path: {{path|}}
- strategy: {{strategy|merge}}
  - preserve_local: {{preserve_local|overwrite}}
- create_backup: {{create_backup|true}}
- dry_run: {{dry_run|true}}

Instrucciones para el agente:
- Explica en español lo que vas a hacer y solicita confirmación si hay acciones destructivas.
- Para `dry_run=true` muestra los comandos propuestos, el conteo de archivos afectados y un diff resumido.
- Si `dry_run=false`, ejecuta los comandos en el orden seguro especificado arriba y devuelve la salida estructurada.

Puntos ambiguos que debes aclarar al usuario
- ¿Deseas conservar cambios locales sin commitear (`stash`) o prefieres abortar si hay cambios? (opciones: `stash`/`abort`/`overwrite`)
- ¿Cuál es la estrategia deseada si hay conflictos? ¿Quieres que intente un `rebase` o prefieres intervención manual?

Notas del mantenedor
- Este prompt está pensado para agentes con permisos para ejecutar comandos Git en el entorno. Si el agente no puede ejecutar comandos (solo asesoría), debe limitarse a listar pasos y comandos a ejecutar manualmente.
